# [Multi-Agent Novel OS] 40년차 작가 전용 AI 소설 집필 시스템 구축 계획

기존 AI의 엉성한 글쓰기와 기억 상실(설정 붕괴)을 완벽하게 극복하기 위해, 깃허브에서 발굴한 최고 수준의 에이전트 프레임워크(Dramatica-flow, Novel-OS, World-forge 등)를 융합하여 **'다중 에이전트 기반 소설 집필 운영체제(Novel OS)'**를 구축하는 아키텍처입니다.

이 시스템은 단일 챗봇이 아닌, **각자의 역할을 가진 4명의 AI 전문가(에이전트)가 서로 협업하고 감시하며 하나의 완벽한 소설을 깎아내는 '디지털 작가 스튜디오'**입니다.

> [!IMPORTANT]
> 이 시스템의 핵심은 '창작'이 아니라 '통제'입니다. AI가 마음대로 글을 쓰는 것을 막고, 작가님이 설계한 '바이블(세계관/캐릭터)'의 법 테두리 안에서만 서사를 전개하도록 강제하는 것이 목표입니다.

## 🛠 시스템 아키텍처: 4-Agent Pipeline

시스템은 4개의 독립된 AI 에이전트와 1개의 상태 관리 코어로 구성됩니다.

### 1. Lore Agent (세계관 관리자)
- **역할:** 마법 체계, 종족 간의 갈등, 지리, 역사적 사건 등을 영구적으로 기억하는 데이터베이스(RAG) 역할을 합니다.
- **동작 방식:** 새로운 챕터가 시작될 때마다, 장소나 등장인물에 관련된 '세계관 규칙'을 자동으로 추출하여 집필자에게 전달합니다. (예: "현재 장소는 북부 왕국이므로, 화염 마법의 위력이 50% 감소한다는 규칙을 적용해야 함")

### 2. Character Architect (캐릭터 심리 분석가)
- **역할:** 엑스트라부터 주인공까지 모든 등장인물의 동기(Motivation), 트라우마, 비밀을 관리합니다.
- **동작 방식:** 뻔한 대사(클리셰)를 방지하기 위해, 특정 사건이 터졌을 때 캐릭터가 '바이블'에 명시된 성격대로 반응하는지 감시합니다. 캐릭터가 갑자기 성격에 맞지 않는 영웅적인 행동을 하려고 하면 즉각 반려합니다.

### 3. Plot Auditor (인과율 검증자 - Dramatica Flow 적용)
- **역할:** 소설의 인과관계와 서사 구조가 무너지지 않도록 방어하는 문지기입니다.
- **동작 방식:** 챕터 초안이 나오면, "이 사건이 이전에 깔아둔 복선과 모순되지 않는가?", "전개 속도(Pacing)가 너무 빠르지 않은가?"를 검증합니다. 모순이 발견되면 집필자(Writer)에게 다시 쓰라고 명령(Reject)합니다.

### 4. Writer Agent (본문 집필자)
- **역할:** 위 3명의 에이전트가 제공한 '완벽한 설정 제약 조건'과 작가님의 '장면 지시(Prompt)'를 바탕으로 실제 소설의 본문(Prose)을 유려하게 작성합니다.
- **동작 방식:** 자유롭게 상상하는 것이 아니라, 철저하게 통제된 조건(제약 기반 생성) 안에서 글을 씁니다.

### 5. Novel-OS State Core (상태 관리 코어)
- 마치 RPG 게임처럼 현재 소설 내의 **시간(Date), 날씨, 캐릭터의 현재 위치, 인벤토리(소지품 상태)**를 전역 변수로 추적합니다. (예: 이전 챕터에서 칼을 잃어버렸는데 다음 챕터에서 칼을 뽑는 오류 원천 차단)

## 🚀 Proposed Changes (구현 로드맵)

승인해주시면, 파이썬을 기반으로 다음 컴포넌트들을 코딩하겠습니다.

### 1. `bible_db.py`
- 작가님의 텍스트 바이블을 쪼개서 벡터 데이터베이스(ChromaDB 또는 FAISS)에 임베딩하는 세계관 메모리 저장소.

### 2. `agent_network.py`
- LangChain 또는 CrewAI를 사용하여 4명의 에이전트(Lore, Character, Auditor, Writer)가 순차적으로 데이터를 주고받으며 토론하는 파이프라인.

### 3. `novel_studio.py`
- 작가님이 다음 챕터의 대략적인 개요("이번 장에서는 주인공이 함정에 빠진다")만 입력하면, 에이전트들이 핑퐁을 치며 검증된 챕터 초안을 출력해주는 메인 실행 파일.

## Verification Plan

1. **바이블 주입 테스트:** 임의의 세계관 규칙("엘프는 거짓말을 하지 못한다")을 입력하고, AI가 거짓말하는 엘프를 등장시키는 오류를 범하는지 테스트합니다.
2. **연속성(Continuity) 테스트:** 챕터 1에서 부상을 입은 상태를 코어 메모리에 저장하고, 챕터 2를 생성할 때 부상의 여파가 글에 자연스럽게(이동 속도 저하 등) 반영되는지 Auditor 에이전트의 작동을 점검합니다.
