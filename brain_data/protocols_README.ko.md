#跨에이전트 통신 프로토콜

프로토콜 모듈은 다중 에이전트 야간 근무의 핵심입니다. 이는 **서로 다른 AI 에이전트가 어떻게 통신하고, 컨텍스트를 공유하고, 작업을 조정하는지**를 정의합니다 — 다른 LLM 엔진에서 실행되더라도.

## 아키텍처 개요

```
┌─────────────┐     night_chat.md      ┌─────────────┐
│  Claude Code │ ◄──────────────────►  │   Gemini     │
│  (개발자)   │                        │  (리서치)   │
└──────┬───────┘                        └──────┬───────┘
       │           ┌─────────────┐             │
       │           │  Heartbeat  │             │
       └──────────►│ (코디네이터)│◄────────────┘
                   └──────┬───────┘
                          │
                   ┌──────▼───────┐
                   │  작업 보드    │
                   │ (Linear/GH)  │
                   └──────────────┘
```

## 통신 채널

### 1. 나이트 채팅 (`night_chat.md`)

에이전트가 실시간 상태 업데이트를 게시하는 공유 마크다운 파일입니다. 파일의 팀 슬랙 채널이라고 생각하세요.

**형식:**
```
[HH:MM] 에이전트이름: 여기에 메시지
[HH:MM] 에이전트이름: 다른 업데이트
```

**규칙:**
- 추가 전용 (기존 메시지 덮어쓰기 안 함)
- 메시지를 간결하게 (1-2줄)
- 시간순 추적을 위해 타임스탬프 포함
- 모든 에이전트가 읽을 수 있음; 각 에이전트가 추가

**사용 사례:**
- 긴 작업 중 진행 상황 업데이트
- 다른 에이전트에 도움 요청
- 발견 또는 경고 공유

### 2. 봇 받은 편지함 (`bot_inbox/`)

JSON 파일을 사용하는 구조화된 메시지 큐입니다. 각 에이전트는 고유한 받은 편지함 디렉토리가 있습니다.

**구조:**
```
bot_inbox/
├── claude/          ← Claude Code 에이전트용 메시지
│   ├── task_001.json
│   ├── msg_002.json
│   └── done/        ← 처리된 메시지
├── gemini/          ← Gemini 에이전트용 메시지
│   └── done/
└── heartbeat/       ← Heartbeat 에이전트용 메시지
    └── done/
```

**메시지 형식:**
```json
{
  "from": "claude",
  "to": "gemini",
  "type": "message",
  "task_id": "TASK-123",
  "message": "Research completed, results in /reports/",
  "ts": "2026-03-11T14:45:00Z"
}
```

**지원 유형:**
- `message` — 자유 형식 통신
- `task` — 작업 할당
- `result` — 작업 완료 보고서

### 3. 알림 스크립트 (`notify.sh`)

다른 에이전트에게 작업 완료를 보고하는 데 사용됩니다.

```bash
./notify.sh <대상 에이전트> <작업_ID> <상태> <요약>
```

**예시:**
```bash
./notify.sh gemini TASK-42 success "auth 모듈 리팩토링 완료, 12개 테스트 통과"
```

### 4. 메시지 스크립트 (`msg.sh`)

에이전트 간 직접 메시지에 사용됩니다.

```bash
./msg.sh <대상 에이전트> "여기에 메시지"
```

## 프로토콜 규칙

1. **추가 전용** — 다른 에이전트의 메시지를 삭제하거나 수정하지 마세요
2. **처리 및 보관** — 처리된 받은 편지함 항목을 `done/`으로 이동
3. **구조화된 것이 자유 형식보다 좋음** — 실행 가능한 항목은 JSON 받은 편지함, FYI 업데이트는 night_chat 사용
4. **멱등 처리** — 에이전트는 중복 메시지를 우아하게 처리해야 합니다
5. **타임아웃 인식** — 메시지가 2개의 하트비트 주기 내에 처리되지 않으면 에스컬레이션

## 작업 보드 통합

프로토콜은 모든 작업 관리 시스템과 함께 작동합니다. 예시:

| 시스템 | 통합 |
|--------|-------------|
| Linear | `linear_tool.py create/update/comment` |
| GitHub Issues | `gh issue create/edit/comment` |
| Jira | Jira CLI 또는 API |
| 일반 파일 | 체크박스 형식의 `tasks.md` |

핵심 원칙: **모든 작업 할당은 보드에 존재해야 합니다**, 채팅에만 있는 것이 아닙니다. 채팅은 조정을 위한 것이고, 보드는 진실의 원천입니다.
