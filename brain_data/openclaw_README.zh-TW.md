# OpenClaw 心跳模組

心跳模組為 AI 代理提供**週期性喚醒系統**。與持續運行的 Claude Code 執行器或基於 cron 的 Gemini 巡邏不同，心跳模式讓代理保持半休眠狀態 — 定期喚醒檢查工作。

## 概念

```
休眠 30分鐘 → 喚醒 → 檢查收件匣 → 處理任務 → 報告 → 休眠
```

這非常適合：
- **協調代理** 向他人派遣工作
- **監控代理** 檢查系統健康
- **路由代理** 根據標籤轉發任務

## 配置

編輯 `heartbeat_config.json` 進行自訂：

| 欄位 | 描述 |
|-------|-------------|
| `interval_minutes` | 代理多久喚醒一次（預設：30） |
| `active_hours` | 心跳的時間窗口（例如：01:00-08:00） |
| `pre_check.collect` | 喚醒代理前要收集的資料來源 |
| `routing.rules` | 如何將任務轉發給其他代理 |
| `reporting.channel` | 發送巡邏報告的位置 |

## 預檢系統

在喚醒 LLM 之前，心跳會自動收集資料：

1. **bot_inbox** — 來自其他代理的待處理訊息和任務
2. **task_board** — 來自任務管理工具的已分配項目
3. **system_health** — 磁碟、記憶體、服務狀態

這通過只呈現相關上下文來減少 LLM Token 使用量。

## 整合

心跳模組與以下整合：
- [跨代理協定](../protocols/) 用於代理間訊息傳遞
- [外掛系統](../plugins/) 用於可擴展的任務處理器
- 任何任務管理工具（Linear、GitHub Issues、Jira 等）

## 適配你的設置

此模組設計為**參考架構**。要實現它：

1. 使用平台的排程系統（cron、systemd timer、cloud scheduler）
2. 編寫預檢腳本收集你的特定資料來源
3. 使用收集的上下文調用你偏好的 LLM
4. 將輸出路由到你的報告通道

請參閱 `heartbeat_config.json` 獲取完整的配置範例。
