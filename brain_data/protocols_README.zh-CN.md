# 跨代理通讯协议

协议模块是多代理夜班的骨干。它定义了**不同的 AI 代理如何通讯、共享上下文和协调工作** — 即使它们运行在不同的 LLM 引擎上。

## 架构概述

```
┌─────────────┐     night_chat.md      ┌─────────────┐
│  Claude Code │ ◄──────────────────►  │   Gemini     │
│  (开发者)   │                        │  (研究者)   │
└──────┬───────┘                        └──────┬───────┘
       │           ┌─────────────┐             │
       │           │  Heartbeat  │             │
       └──────────►│ (协调者)    │◄────────────┘
                   └──────┬───────┘
                          │
                   ┌──────▼───────┐
                   │  任务面板     │
                   │ (Linear/GH)  │
                   └──────────────┘
```

## 通讯通道

### 1. 夜聊 (`night_chat.md`)

代理发布实时状态更新的共享 Markdown 文件。把它想象成文件中的团队 Slack 频道。

**格式：**
```
[HH:MM] 代理名称: 讯息内容
[HH:MM] 代理名称: 另一个更新
```

**规则：**
- 仅附加（从不覆盖现有讯息）
- 保持讯息简洁（1-2 行）
- 包含时间戳以进行时间顺序追踪
- 任何代理都可以阅读；每个代理附加讯息

**用例：**
- 长时间任务期间的进度更新
- 向其他代理请求协助
- 分享发现或警告

### 2. 机器人收件箱 (`bot_inbox/`)

使用 JSON 文件的结构化讯息队列。每个代理都有自己的收件箱目录。

**结构：**
```
bot_inbox/
├── claude/          ← Claude Code 代理的讯息
│   ├── task_001.json
│   ├── msg_002.json
│   └── done/        ← 已处理的讯息
├── gemini/          ← Gemini 代理的讯息
│   └── done/
└── heartbeat/       ← Heartbeat 代理的讯息
    └── done/
```

**讯息格式：**
```json
{
  "from": "claude",
  "to": "gemini",
  "type": "message",
  "task_id": "TASK-123",
  "message": "Research completed, results in /reports/",
  "ts": "2026-03-11T14:45:00Z"
}
```

**支援的类型：**
- `message` — 自由格式通讯
- `task` — 工作分配
- `result` — 任务完成报告

### 3. 通知脚本 (`notify.sh`)

用于向其他代理报告任务完成。

```bash
./notify.sh <目标代理> <任务ID> <状态> <摘要>
```

**范例：**
```bash
./notify.sh gemini TASK-42 success "重构了 auth 模块，12 个测试通过"
```

### 4. 讯息脚本 (`msg.sh`)

用于代理之间的直接讯息。

```bash
./msg.sh <目标代理> "你的讯息内容"
```

## 协议规则

1. **仅附加** — 从不删除或修改其他代理的讯息
2. **处理和归档** — 将处理的收件箱项目移到 `done/`
3. **结构化优于自由格式** — 对于可操作的项目首选 JSON 收件箱，夜聊用于 FYI 更新
4. **幂等处理** — 代理应该优雅地处理重复讯息
5. **超时感知** — 如果讯息在 2 个心跳周期内未处理，则升级

## 任务面板整合

协议与任何任务管理系统配合使用。范例：

| 系统 | 整合 |
|--------|-------------|
| Linear | `linear_tool.py create/update/comment` |
| GitHub Issues | `gh issue create/edit/comment` |
| Jira | Jira CLI 或 API |
| 纯文件 | 带有勾选格式的 `tasks.md` |

关键原则：**每个任务分配都应该存在于面板上**，而不只是在聊天中。聊天用于协调；面板是真相来源。
